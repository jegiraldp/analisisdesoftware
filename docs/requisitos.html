1. Requisitos Funcionales (RF)

Definen lo que el sistema debe hacer. Describen las funciones, servicios o tareas específicas.

Buenas Prácticas

  • Usar lenguaje claro: "El sistema debe [acción]".
  • Ser atómico: Un requisito por frase.
  • Evitar ambigüedades (palabras como "rápido" o "fácil").
Ejemplos:
  1. El sistema permitirá al usuario recuperar su contraseña mediante correo electrónico.
  2. El sistema generará un reporte mensual de ventas en formato PDF.
  3. El sistema debe validar que el stock sea mayor a cero antes de confirmar una compra.

2. Requisitos No Funcionales (RNF)

Definen cómo debe funcionar el sistema (restricciones o cualidades de calidad).

Buenas Prácticas

  • Deben ser medibles (verificables).
  • Relacionarlos con atributos de calidad (seguridad, rendimiento, usabilidad).
  • Definir rangos aceptables.
Ejemplos:
  1. Rendimiento: Las consultas a la base de datos deben responder en menos de 2 segundos.
  2. Seguridad: Todas las contraseñas deben estar encriptadas bajo el estándar AES-256.
  3. Disponibilidad: El sistema debe estar operativo el 99.9% del tiempo anual.

3. Historias de Usuario (HU)

Breve descripción de una funcionalidad contada desde la perspectiva del usuario.

Buenas Prácticas (Modelo INVEST)

  • Independiente, Negociable, Valiosa, Estimalble, Small (Pequeña), Testeable.
  • Utilizar la estructura: "Como [rol], quiero [acción], para [beneficio]".
Ejemplos:
  1. Como Cliente, quiero filtrar productos por precio, para encontrar lo que se ajuste a mi presupuesto.
  2. Como Administrador, quiero suspender cuentas de usuario, para mantener la seguridad de la plataforma.
  3. Como Vendedor, quiero ver un historial de mis ventas, para llevar control de mis comisiones.

Plantilla de Documentación de HU

HU-05: Filtrado de Productos
Como: Cliente Quiero: Filtrar por categorías
Para: No perder tiempo viendo productos que no me interesan.
Criterios de Aceptación:
- El sistema muestra un menú lateral con las categorías disponibles.
- Al seleccionar una categoría, la lista de productos se actualiza instantáneamente.

4. Tarjetas CRC (Class, Responsibility, Collaboration)

Técnica de diseño orientado a objetos para identificar clases y sus interacciones.

Buenas Prácticas

  • Mantener las responsabilidades a un alto nivel (no métodos detallados).
  • Si una clase tiene demasiadas responsabilidades, considerar dividirla.
  • Realizar sesiones grupales de "role-playing" para validar flujos.
Ejemplos de Clases:
  1. Pedido: Calcula el total, valida stock. Colabora con Producto y Cliente.
  2. Carrito: Almacena ítems temporalmente. Colabora con Producto.
  3. Usuario: Gestiona perfil y autenticación.
